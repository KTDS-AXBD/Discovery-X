## 1. 요약 (Executive Summary)

**한 줄 정의:**
Discovery-X 앱 내에서 대화형 인터뷰 → PRD 자동 생성 → AI 다중 검토 → 피드백 반영 → 착수 판단까지 수행하는 웹 UI 기능.

**배경:**
현재 요구사항 기획은 Claude Code CLI 전용 스킬로만 가능하여 비개발자 접근 불가. Discovery-X의 아이디어/제안 흐름과 분리되어 비효율적.

**목표:**
- 비개발자를 포함한 팀 전체가 웹 UI에서 PRD 기획/검토 가능
- 아이디어 → PRD → 사업제안 흐름이 하나의 앱에서 완결
- CLI 스킬의 검토 품질을 웹에서도 동일하게 제공

## 2. 문제 정의

### 2.1 현재 상태 (As-Is)
- CLI 전용 → 비개발자(5명 중 4명) 접근 불가
- 로컬 Node 스크립트 실행 필요 → 검토 접근성 낮음
- 아이디어 분석과 PRD 기획이 별개 도구
- 검토 결과가 로컬 파일로만 존재

### 2.2 목표 상태 (To-Be)
- 웹 UI 인터뷰 → PRD 생성 → AI 검토 → 피드백 → 착수 판단 (원스톱)
- 아이디어 데이터 → PRD 초안 자동 반영, 완성 PRD → 사업제안 참조

### 2.3 시급성
- 운영 실험 진행 중, 비개발자 기획 참여 수단 부재
- 인터뷰 스킬 사용자가 개발자 1명으로 제한

## 3. 사용자 및 이해관계자

### 3.1 주 사용자
| 구분 | 설명 | 주요 니즈 |
|------|------|-----------|
| BD팀 비개발자 | 사업 기획/분석 | 웹에서 PRD 기획 수행, 가이드 제공 |
| BD팀 개발자 | 개발/운영 | 아이디어→PRD→제안 앱 내 완결 |

### 3.2 핵심 사용자 여정

**여정 A: 아이디어 → PRD (비개발자)**
```
아이디어 상세 → "PRD 작성" → 온보딩 가이드(최초 1회)
→ 인터뷰 화면 (분석 데이터 사전 로드, 진행률 표시)
→ 채팅형 5파트 인터뷰 (예시 답변 접힘 제공)
→ 인터뷰 완료 → 요약 확인 → PRD v1 자동 생성
→ PRD 미리보기/편집 → "AI 검토" (외부 전송 고지)
→ 검토 로딩 (모델별 진행 뱃지) → 피드백 뷰어 + 스코어카드
→ 수정 → 재검토 또는 착수 확정 (만족도 1문항)
```

### 3.3 비개발자 UX 설계

- **온보딩 가이드**: 최초 접근 시 3단계 스포트라이트 (기존 온보딩 패턴 재사용) — "인터뷰란?", "AI가 검토해요", "스코어로 판단"
- **예시 답변**: 파트별 접힘 형태 예시 ("예: '수동 엑셀 관리에 30분씩 걸려요'")
- **진행률**: 5파트 프로그레스 바 + 현재 파트명
- **중간 저장**: DB 저장 → 브라우저 닫아도 재진입 시 이어하기
- **도움말**: 화면 하단 "도움이 필요하세요?" FAQ 링크

### 3.4 이탈 분석 설계

인터뷰 완주율 진단을 위해 단계별 이벤트를 기록:

| 이벤트 | 시점 | 기록 내용 |
|--------|------|-----------|
| `interview_start` | 인터뷰 시작 | userId, prdId, sourceType(idea/standalone) |
| `interview_part_complete` | 각 파트 완료 | partNumber(1~5), elapsedSeconds |
| `interview_abandon` | 인터뷰 이탈 | lastPart, totalElapsed |
| `prd_generated` | PRD 생성 | version, wordCount |
| `review_start` | 검토 시작 | models[], roundNumber |
| `review_complete` | 검토 완료 | score, verdict |
| `prd_finalized` | 착수 확정 | finalScore, satisfactionRating |

> 기존 `cron_logs` 테이블 활용 또는 `prd_events` 전용 테이블 생성

### 6.2 기술 리스크 + 장애 대응

| 리스크 | 대응 | Fallback |
|--------|------|----------|
| AI 30초 타임아웃 | `Promise.allSettled` + 25초 개별 타임아웃 | 1개만 성공해도 피드백 표시 |
| AI API 장애 | 4단계 fallback (OpenAI→Gemini→DeepSeek→수동) | radar-worker 패턴 재사용 |
| SSE 끊김 | 폴링 fallback 30초 간격 | `/api/prd-studio/review/:id/status` |
| 비용 초과 | 월간 추적 + 80% 알림 | cost 모듈 budget_policies |

### 6.3 테스트/QA

### 6.4 모니터링/로깅

- **이탈 분석**: §3.4 이벤트 기록 → 주간 요약 (완주율, 이탈 파트 분포)
- **비용 추적**: `usage_events` 테이블 (기존 cost 모듈)
- **에러 대시보드**: `/admin/monitoring` PRD Studio 섹션 (API 성공률, 응답 시간)
- **알림**: 비용 80% 임계치 + API 연속 3회 실패 시 관리자 알림

### 6.5 CLI ↔ 웹 호환성 정책

| 항목 | 방침 |
|------|------|
| PRD 포맷 | 동일한 마크다운 템플릿 사용 (prd-template.md) |
| 데이터 이관 | CLI PRD는 웹에서 import 불필요 — 독립 운영 (기존 CLI PRD는 `docs/` 로컬 파일로 유지) |
| 검토 결과 | 웹: DB 저장 / CLI: 로컬 파일 — 상호 참조 불필요 |
| 방향 | CLI 스킬은 개발자 전용 유지, 웹 UI는 팀 전체 대상 — 병행 운영 |

> CLI와 웹을 통합하지 않고 독립 채널로 병행. 팀 전체 공유가 필요한 PRD는 웹에서 작성.

### 6.6 민감 정보 + 보안 정책

- **전송 전 고지**: "PRD 내용이 외부 AI(OpenAI, Google)로 전송됩니다" 확인 다이얼로그
- **작성 지침**: 인터뷰 시작 화면에 "개인정보, 인사정보, 미공개 재무정보 포함 금지" 명시
- **기술 보호**: API 키 서버사이드 전용 / 원본 PRD 별도 외부 로깅 안 함
- **데이터 보관**: PRD 및 검토 결과는 D1 DB에 저장, 삭제 시 관련 검토 결과도 cascade 삭제
- **유출 대응**: 민감 정보 포함 PRD가 외부 전송된 경우 → 관리자 알림 + 해당 PRD 검토 기록 삭제 + 재발 방지 조치 문서화
- **인증**: 기존 Google OAuth + 세션, requireUser() 가드 적용

### 6.7 인력/예산
- 투입: 1명 (Sinclair) — 장애 Escalation: BD팀 리더
- AI API: PRD당 ~$0.05 (review), ~$0.20 (sixhats), 월 $30 미만
