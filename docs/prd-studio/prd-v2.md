**작성자:** AX BD팀

## 1. 요약 (Executive Summary)

**한 줄 정의:**
Discovery-X 앱 내에서 대화형 인터뷰 → PRD 자동 생성 → AI 다중 검토 → 피드백 반영 → 착수 판단까지 수행하는 웹 UI 기능.

**배경:**
현재 요구사항 기획은 Claude Code CLI 전용 스킬(`ax-14-req-interview`)로만 가능하다. 비개발자 팀원은 CLI에 접근할 수 없어 기획 프로세스에 참여하지 못하고, 개발자도 CLI와 웹 앱 사이를 오가며 비효율적으로 작업한다. Discovery-X의 아이디어/사업제안 흐름과 요구사항 기획이 완전히 분리되어 있어 컨텍스트 전환 비용이 높다.

**목표:**
- 비개발자를 포함한 팀 전체가 웹 UI에서 PRD 기획/검토 가능
- 아이디어 → PRD → 사업제안 흐름이 하나의 앱에서 완결
- CLI 스킬의 검토 품질(다중 AI 검토, 스코어카드)을 웹에서도 동일하게 제공

## 2. 문제 정의

### 2.1 현재 상태 (As-Is)
- 인터뷰 스킬은 Claude Code CLI(`/ax-14-req-interview`) 전용
- 비개발자 팀원(5명 중 4명)은 CLI 환경에 접근 불가
- PRD 검토는 `review-api.mjs` Node 스크립트를 로컬에서 실행해야 함
- 아이디어 분석 결과와 PRD 기획이 별개 도구에서 관리됨
- 검토 결과(스코어카드, 피드백)가 로컬 파일로만 존재

### 2.2 목표 상태 (To-Be)
- 웹 UI에서 대화형 인터뷰 진행 → PRD 자동 생성
- 생성된 PRD에 대해 AI 다중 검토를 앱 내에서 실행
- 검토 피드백 → PRD 수정 → 재검토 사이클을 반복
- 스코어카드 자동 채점 → 착수 판단 지원
- 아이디어 분석 데이터를 PRD 초안에 자동 반영
- 완성된 PRD를 사업제안에서 참조

### 2.3 시급성
- 운영 실험(2026-01-31~) 진행 중이나 비개발자 기획 참여 수단 부재
- 수동 기획 과정에서 맥락 손실 및 반복 작업 발생
- 이 기능이 없으면 인터뷰 스킬의 사용자 범위가 개발자 1명으로 제한

## 3. 사용자 및 이해관계자

### 3.1 주 사용자
| 구분 | 설명 | 주요 니즈 |
|------|------|-----------|
| BD팀 비개발자 | 사업 기획/분석 담당 | CLI 없이 웹에서 PRD 기획 및 검토 수행 |
| BD팀 개발자 | Discovery-X 개발/운영 | 아이디어→PRD→제안 흐름을 앱 내에서 완결 |

### 3.2 이해관계자
| 구분 | 역할 | 영향도 |
|------|------|--------|
| Sinclair | 개발자, 프로젝트 오너 | 높음 |
| BD팀 리더 | 착수 의사결정 | 높음 |

### 3.3 사용 환경
- 기기: PC (데스크톱 웹 브라우저)
- 네트워크: 인터넷 (Cloudflare Pages)
- 기술 수준: 혼합 (개발자 + 비개발자, AI 챗봇 경험 있음)

### 3.4 핵심 사용자 여정 (Use Case)

**여정 A: 아이디어 → PRD 작성 (비개발자)**
```
아이디어 상세 페이지 → "PRD 작성" 버튼 클릭
→ PRD Studio 인터뷰 화면 진입 (아이디어 분석 데이터 사전 로드)
→ 채팅형 인터뷰 5파트 진행 (파트별 질문 1개씩, 음성 입력 가능)
→ 인터뷰 완료 시 요약 확인 → PRD v1 자동 생성
→ PRD 미리보기 + 편집 → "AI 검토 요청" 버튼
→ 검토 진행 중 로딩 UI (모델별 진행 상태 표시)
→ 검토 완료: 피드백 뷰어 (AI별 탭) + 스코어카드
→ 피드백 반영하여 PRD 수정 → 재검토 또는 착수 확정
```

**여정 B: 독립 PRD 작성 (개발자/비개발자)**
```
GNB "실험실" → PRD Studio 탭 → "새 PRD 작성" 버튼
→ 프로젝트명 입력 → 인터뷰 시작 → (여정 A와 동일 흐름)
```

**여정 C: 기존 PRD 재검토**
```
PRD 목록에서 기존 PRD 선택 → PRD 편집 → "재검토" 버튼
→ Round 2+ 검토 실행 → 이전 라운드 대비 개선도 표시
```

## 5. 성공 기준

### 5.1 정량 지표 (KPI)
| 지표 | 현재값 | 목표값 | 측정 방법 |
|------|--------|--------|-----------|
| 비개발자 PRD 작성 | 0건 | 1건 이상/월 | prds 테이블 createdBy 확인 |
| PRD 검토 완주율 | 50% (CLI) | 90%+ | 검토 라운드 완료 비율 |

| 검토 품질 만족도 | [미측정] | 긍정 70%+ | 착수 확정 시 "검토 유용했나요?" 1문항 |

### 5.2 MVP 최소 기준
- [ ] 웹 UI에서 5파트 인터뷰 진행 가능
- [ ] 인터뷰 결과로 PRD v1 자동 생성
- [ ] 최소 2개 AI 모델로 검토 실행 가능
- [ ] 피드백 결과를 구조화하여 표시

## 6. 제약 조건

### 6.1 일정
- **Phase 1** (DB + API): ~3일 — 스키마 설계, AI 호출 서비스, 검토 API
- **Phase 2** (인터뷰 UI): ~3일 — 채팅 인터페이스, 인터뷰 진행, PRD 생성
- **Phase 3** (검토 + 스코어카드): ~2일 — 피드백 뷰어, 스코어카드, PRD 편집
- **Phase 4** (연동 + 마무리): ~2일 — 아이디어/제안 연동, 테스트, 배포
- 총 예상: **~10일** (1인 개발 기준, 리스크 버퍼 포함)

### 6.2 기술 스택
- 프론트엔드: React 19 + Tailwind CSS 4 + @axis-ds
- 백엔드: Remix v2 (Cloudflare Pages Edge Runtime)
- DB: D1 (SQLite) + Drizzle ORM
- AI 호출: Edge Runtime에서 직접 `fetch` (Node child_process 불가)
- 기존 패턴: SSE 스트리밍 (api.chat.ts, api.ideas.$id.analyze.ts 참조)

### 6.3 기술 리스크 대응 (PoC 계획)

| 리스크 | 영향 | PoC/대응 |
|--------|------|----------|
| Edge Runtime AI 병렬 호출 타임아웃 | 검토 API 30초 초과 시 실패 | `Promise.allSettled` + 개별 타임아웃(25초) + 부분 성공 허용 |
| D1 동시 쓰기 경합 | 병렬 검토 결과 저장 시 충돌 | 검토 결과를 JSON 컬럼 1개에 저장 (개별 INSERT 불필요) |
| SSE 스트리밍 도중 연결 끊김 | 검토 진행 상태 유실 | 폴링 fallback: `/api/prd-studio/review/:id/status` |

> Phase 1 착수 시 AI 병렬 호출 PoC를 먼저 수행하여 타임아웃 검증

### 6.4 인력/예산
- 투입 가능 인원: 1명 (Sinclair)
- AI API 예산: PRD당 $1 이하 (월 $30 미만 목표)
- 비용 추정: review 모드 ~$0.05/회 (2모델), sixhats ~$0.20/회 (20턴)

### 6.5 컴플라이언스 / 민감 정보 정책

**원칙: PRD 내용 전체가 AI API로 전송된다는 것을 사용자가 인지해야 한다.**

- **전송 전 고지**: 검토 실행 버튼에 "PRD 내용이 외부 AI 서비스(OpenAI, Google)로 전송됩니다" 안내 문구 표시
- **민감 정보 지침**: PRD 작성 가이드에 "고객 개인정보, 내부 인사정보, 미공개 재무정보는 포함하지 마세요" 명시
- **기술적 보호**: API 키는 서버사이드(`context.cloudflare.env`)에서만 사용, 클라이언트 노출 없음
- **로그 관리**: AI 응답은 DB에 저장하되, 원본 PRD 텍스트의 별도 로깅은 하지 않음
- **기존 인증**: Discovery-X Google OAuth + 세션 기반 접근 제어 준수
