# Design: ontology-intelligence (Phase 1 — 데이터 형성 자동화)

> **Feature**: 온톨로지 기반 관계 분석 + 미래 예측 시스템
> **Author**: Claude
> **Created**: 2026-02-11
> **Status**: Draft
> **Plan**: `docs/01-plan/features/ontology-intelligence.plan.md`

---

## 1. 설계 범위

이 Design 문서는 **Phase 1: 온톨로지 데이터 형성 자동화**를 다룬다.
Phase 2(관계 분석)와 Phase 3(미래 예측)은 Phase 1 완료 후 별도 Design으로 진행한다.

### Phase 1 목표
- Evidence 생성 시 **자동으로 엔티티 추출** (LLM 기반 NER + 분류)
- 추출된 엔티티 간 **관계 자동 추론** (관계 타입 + 강도)
- Discovery 간 **공유 엔티티 자동 감지** 및 교차 링크
- 추출 결과 **사용자 검토 큐** (Human-in-the-Loop)

## 2. 현행 시스템 분석

### 2.1 기존 온톨로지 스키마 (5 테이블)

| 테이블 | 역할 | 현재 상태 |
|--------|------|----------|
| `ontologyTypes` | 10개 고정 엔티티 타입 정의 | ✅ 시드 데이터 존재 |
| `contextNodes` | Discovery별 엔티티 노드 | ⚠️ 수동 생성만 가능, 대부분 비어있음 |
| `contextEdges` | 노드 간 관계 | ⚠️ 수동 생성만 가능 |
| `contextSnapshots` | 단계별 그래프 스냅샷 | ✅ 동작 중 |
| `evidenceDuplicateCandidates` | 근거 중복 후보 | ✅ embeddings cron에서 자동 생성 |

### 2.2 기존 Agent 도구 (5개)

| 도구 | 자율도 | 용도 |
|------|--------|------|
| `extractEntities` | 3 | 엔티티 추출 → 노드 생성 |
| `linkEntities` | 3 | 노드 간 관계 생성 |
| `queryGraph` | 1 | 그래프 조회 + 통계 |
| `getDuplicateQueue` | 1 | 중복 후보 조회 |
| `reviewDuplicate` | 2 | 중복 결정(병합/무시) |

### 2.3 기존 Embedding Sync Cron 패턴

```
api.cron.embeddings.ts
  ├─ CRON_SECRET 인증
  ├─ 모든 활성 테넌트 순회
  │   ├─ stale 레코드 감지 (embeddingUpdatedAt IS NULL OR < updatedAt)
  │   ├─ OpenAI embedding 생성
  │   ├─ Vectorize 업서트
  │   └─ 중복 후보 생성
  └─ 결과 JSON 반환
```

### 2.4 핵심 제약사항

| 제약 | 영향 | 대응 |
|------|------|------|
| `contextNodes.discoveryId` NOT NULL FK | 글로벌 엔티티 표현 불가 | `globalEntityId` 추가 |
| Agent 도구 자율도 3 (수동 호출) | 자동 파이프라인 없음 | Cron 엔드포인트 신규 |
| 온톨로지 타입 10개 고정 | 유연성 제한 | 우선 10개 활용, 추후 확장 |
| Edge strength 0-100 정수 | 확률 계산 시 정밀도 | API에서 0-1 변환 (기존 패턴) |

## 3. 스키마 변경

### 3.1 evidence 테이블 — 추출 추적 컬럼

```sql
-- 기존 evidence 테이블에 추가
ALTER TABLE evidence ADD COLUMN ontology_extracted_at INTEGER;
```

**역할**: `embeddingUpdatedAt`과 동일 패턴. Cron이 "이미 처리된 Evidence"를 건너뛰기 위한 마커.

| 컬럼 | 타입 | 기본값 | 설명 |
|------|------|--------|------|
| `ontologyExtractedAt` | integer (timestamp) | NULL | 온톨로지 추출 완료 시점 |

**Stale 감지 로직**:
```sql
WHERE ontology_extracted_at IS NULL
   OR ontology_extracted_at < created_at
```

### 3.2 contextNodes 테이블 — 글로벌 엔티티 ID

```sql
-- 기존 contextNodes 테이블에 추가
ALTER TABLE context_nodes ADD COLUMN global_entity_id TEXT;
CREATE INDEX idx_context_nodes_global_entity ON context_nodes(global_entity_id);
```

| 컬럼 | 타입 | 기본값 | 설명 |
|------|------|--------|------|
| `globalEntityId` | text | NULL | 같은 개념의 노드를 그룹핑하는 ID |

**사용 패턴**:
- 신규 엔티티 추출 시 기존 노드와 Embedding 유사도 비교
- 유사도 ≥ 0.85 → 기존 노드의 `globalEntityId`를 공유
- 유사도 < 0.85 → 새 `globalEntityId` 생성 (crypto.randomUUID())
- 교차 Discovery 그래프 조회: `WHERE globalEntityId = ?`

### 3.3 contextNodes 테이블 — 신뢰도 + 자동생성 마커

```sql
ALTER TABLE context_nodes ADD COLUMN confidence REAL DEFAULT 1.0;
ALTER TABLE context_nodes ADD COLUMN auto_generated INTEGER DEFAULT 0;
ALTER TABLE context_nodes ADD COLUMN reviewed INTEGER DEFAULT 0;
```

| 컬럼 | 타입 | 기본값 | 설명 |
|------|------|--------|------|
| `confidence` | real | 1.0 | LLM 추출 신뢰도 (0.0~1.0) |
| `autoGenerated` | integer | 0 | 0=수동, 1=자동추출 |
| `reviewed` | integer | 0 | 0=미검토, 1=승인, 2=거절 |

### 3.4 contextEdges 테이블 — 동일 확장

```sql
ALTER TABLE context_edges ADD COLUMN confidence REAL DEFAULT 1.0;
ALTER TABLE context_edges ADD COLUMN auto_generated INTEGER DEFAULT 0;
ALTER TABLE context_edges ADD COLUMN reviewed INTEGER DEFAULT 0;
```

## 4. LLM 엔티티 추출 엔진

### 4.1 모듈 구조

**파일**: `app/lib/ontology/extractor.ts`

```
extractOntologyFromEvidence(db, env, evidence, discoveryId)
  ├─ buildExtractionPrompt(evidence, existingNodes)
  ├─ callLLM(prompt) → structured JSON
  ├─ matchGlobalEntities(newEntities, env) → globalEntityId 매칭
  ├─ createNodes(db, entities) → contextNodes INSERT
  ├─ createEdges(db, relations) → contextEdges INSERT
  └─ return { nodesCreated, edgesCreated, errors }
```

### 4.2 LLM 프롬프트 설계

**입력**:
```json
{
  "evidence": {
    "content": "국내 ESG 시장 규모가 2025년 150조원으로 성장...",
    "type": "DATA",
    "strength": "B",
    "reliabilityLabel": "reported"
  },
  "existingNodes": [
    { "label": "ESG 시장", "type": "market", "globalEntityId": "ge-001" },
    { "label": "탄소중립 정책", "type": "strategy", "globalEntityId": "ge-002" }
  ],
  "ontologyTypes": [
    { "id": "ONT-01", "nameKo": "고객 세그먼트", "domain": "customer" },
    // ... 10개
  ]
}
```

**출력 (structured JSON)**:
```json
{
  "entities": [
    {
      "label": "ESG 시장",
      "ontologyTypeId": "ONT-02",
      "confidence": 0.95,
      "isExisting": true,
      "matchedGlobalEntityId": "ge-001"
    },
    {
      "label": "ESG 펀드 투자자",
      "ontologyTypeId": "ONT-01",
      "confidence": 0.82,
      "isExisting": false,
      "matchedGlobalEntityId": null
    }
  ],
  "relations": [
    {
      "fromLabel": "ESG 시장",
      "toLabel": "ESG 펀드 투자자",
      "relationType": "relates_to",
      "strength": 0.75,
      "confidence": 0.80
    }
  ]
}
```

### 4.3 LLM 호출 상세

| 항목 | 설정 |
|------|------|
| **모델** | Claude 3.5 Haiku (비용 최적화) |
| **Temperature** | 0.1 (일관성 우선) |
| **Max Tokens** | 2048 |
| **응답 형식** | JSON (tool_use 또는 system prompt 강제) |
| **예상 비용** | Evidence 1건당 ~$0.002-0.005 |
| **타임아웃** | 30초 |
| **재시도** | 최대 2회 (JSON 파싱 실패 시) |

### 4.4 신뢰도 필터링

| 신뢰도 | 처리 |
|--------|------|
| ≥ 0.8 | 자동 생성 + 검토 큐 등록 (reviewed=0) |
| 0.5 ~ 0.8 | 검토 큐에만 등록 (자동 생성 안 함) |
| < 0.5 | 무시 (로그만 남김) |

## 5. 글로벌 엔티티 매칭

### 5.1 매칭 전략

```
신규 엔티티 "ESG 시장"
    ↓
[1단계: 정확 매칭] — label 동일 + ontologyTypeId 동일
    → 매칭 성공 시 기존 globalEntityId 공유
    ↓ 매칭 실패
[2단계: Embedding 유사도] — Vectorize 검색
    → cosine similarity ≥ 0.85 + 같은 ontologyTypeId
    → 매칭 성공 시 기존 globalEntityId 공유
    ↓ 매칭 실패
[3단계: 신규 생성] — crypto.randomUUID()
```

### 5.2 Embedding 생성

기존 Evidence embedding과 동일한 OpenAI `text-embedding-3-small` 사용.
새 Vectorize 인덱스는 만들지 않고, **엔티티 label을 Evidence 검색에 활용**하는 방식으로 간소화.

```typescript
// 엔티티 label로 기존 contextNodes에서 유사 검색
const embedding = await generateEmbedding(apiKey, entity.label);
const candidates = await db
  .select()
  .from(contextNodes)
  .where(
    and(
      eq(contextNodes.ontologyTypeId, entity.ontologyTypeId),
      isNotNull(contextNodes.globalEntityId)
    )
  );
// candidates의 label 임베딩과 코사인 유사도 비교
```

**간소화 이유**: Phase 1에서는 엔티티 수가 적으므로 DB 직접 검색 + JS 유사도 계산으로 충분. 노드 수 1000개 이상 시 Vectorize 인덱스 추가 검토.

### 5.3 대안: 레이블 기반 간단 매칭 (MVP)

Phase 1 MVP에서는 Embedding 유사도 대신 **정규화 레이블 매칭**으로 시작:

```typescript
function normalizeLabel(label: string): string {
  return label
    .toLowerCase()
    .replace(/\s+/g, " ")
    .trim();
}

// 정확 매칭: normalize(existingLabel) === normalize(newLabel) && sameType
```

**이점**: 외부 API 호출 없이 빠른 매칭. 오탐 최소화.
**한계**: "ESG 시장" ↔ "ESG Market" 같은 다국어 변형 미감지.

→ **MVP는 정규화 레이블 매칭, 이후 Embedding 유사도 추가** 권장.

## 6. Cron 엔드포인트

### 6.1 자동 추출 Cron

**파일**: `app/routes/api.cron.ontology-extract.ts`

```typescript
// 패턴: api.cron.embeddings.ts 재사용
export async function loader({ request, context }: LoaderFunctionArgs) {
  // 1. CRON_SECRET 인증
  const url = new URL(request.url);
  const secret = url.searchParams.get("secret");
  if (secret !== cronSecret) return json({ error: "Unauthorized" }, 401);

  // 2. 환경변수 검증
  const anthropicKey = getEnvVar(context, "ANTHROPIC_API_KEY");
  if (!anthropicKey) return json({ error: "Missing ANTHROPIC_API_KEY" }, 500);

  // 3. 테넌트별 처리
  const results = [];
  for (const tenant of activeTenants) {
    const result = await extractOntologyBatch(db, env, anthropicKey, {
      tenantId: tenant.id,
      batchSize: 5,              // Evidence 5건씩
      confidenceThreshold: 0.5,  // 0.5 미만 무시
    });
    results.push({ tenantId: tenant.id, ...result });
  }

  return json({ success: true, results });
}
```

### 6.2 배치 처리 로직

```typescript
async function extractOntologyBatch(db, env, apiKey, options) {
  const { tenantId, batchSize, confidenceThreshold } = options;

  // 1. Stale evidence 조회
  const staleEvidence = await db
    .select()
    .from(evidence)
    .innerJoin(discoveries, eq(evidence.discoveryId, discoveries.id))
    .where(
      and(
        eq(discoveries.tenantId, tenantId),
        or(
          isNull(evidence.ontologyExtractedAt),
          sql`${evidence.ontologyExtractedAt} < ${evidence.createdAt}`
        )
      )
    )
    .limit(batchSize);

  // 2. Discovery별 기존 노드 조회 (매칭용)
  // 3. 각 Evidence에 대해 LLM 추출
  // 4. 글로벌 엔티티 매칭
  // 5. 노드/엣지 생성
  // 6. ontologyExtractedAt 업데이트
  // 7. 결과 반환

  return {
    evidenceProcessed: number,
    nodesCreated: number,
    edgesCreated: number,
    globalEntitiesMatched: number,
    errors: string[]
  };
}
```

## 7. 검토 큐 (Human-in-the-Loop)

### 7.1 기존 getDuplicateQueue 패턴 재사용

`ontology-tools.ts`의 `getDuplicateQueue` 도구가 이미 검토 큐 패턴을 구현.
동일 패턴으로 **자동 생성된 엔티티/관계 검토** 큐를 추가.

### 7.2 검토 API

**파일**: `app/routes/api.ontology.review.ts`

```
POST /api/ontology/review
Body: {
  nodeId: string;      // 또는 edgeId
  type: "node" | "edge";
  action: "approve" | "reject" | "edit";
  editedLabel?: string;  // action=edit 시
  editedTypeId?: string; // action=edit 시
}
```

| 액션 | 동작 |
|------|------|
| `approve` | `reviewed = 1` (승인) |
| `reject` | `reviewed = 2` (거절) — 노드/엣지 비활성화 |
| `edit` | `reviewed = 1` + label/typeId 수정 |

### 7.3 검토 UI

기존 `evidence.duplicates.tsx` (DuplicateCard) 패턴을 재사용하여
`ontology.review.tsx` 라우트에 검토 카드 리스트를 구현.

## 11. 에러 처리

| 상황 | 처리 | 복구 방법 |
|------|------|----------|
| LLM 응답 JSON 파싱 실패 | 2회 재시도 → 건너뛰기 | 다음 Cron에서 재처리 |
| LLM API 타임아웃 | 30초 후 건너뛰기 | 다음 Cron에서 재처리 |
| 글로벌 엔티티 매칭 실패 | 새 globalEntityId 생성 | 수동 머지 가능 |
| 노드 중복 생성 | label + discoveryId 기준 UPSERT | 기존 노드 업데이트 |
| CRON_SECRET 불일치 | 401 반환 | 환경변수 확인 |
| ANTHROPIC_API_KEY 미설정 | 500 반환, 처리 중단 | 환경변수 추가 |
