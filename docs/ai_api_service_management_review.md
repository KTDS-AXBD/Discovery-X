# AI API 서비스 관리 개발계획 검토의견

## 총평

이 계획은 **“AI API 운영을 단순 연결이 아니라 비용·예산·품질·공급자 라우팅까지 통합 통제하는 플랫폼”** 으로 보겠다는 점에서 방향성은 맞습니다. 특히 기존 구현과의 GAP을 기준으로 요구사항을 구조화한 점은 실무적으로 괜찮습니다. 다만 현재 안은 **요구사항 수준에서는 설득력 있지만, 운영정책·데이터 기준·제어 우선순위가 아직 덜 잠겨 있어서 그대로 개발에 들어가면 복잡도와 정책 충돌이 커질 가능성**이 있습니다.

평가를 한 줄로 하면:

**전략적으로는 타당, 제품적으로는 매력적, 하지만 개발계획으로는 아직 “과제 목록”에 가깝고 “실행 설계”로는 한 단계 부족**합니다.

점수로 보면 대략 이렇게 보입니다.

- **방향성 적합성:** 8.5/10  
- **운영 실효성:** 7.5/10  
- **개발 난이도 대비 우선순위 정합성:** 6.5/10  
- **정산/통제 정책의 완성도:** 6/10  
- **전체 실행 가능성:** 7/10

## 1. 잘 잡은 부분

### 1) 비용-예산-API 라우팅을 분리하지 않고 하나의 운영체계로 본 점

좋습니다. 많은 시스템이 “사용량 집계”, “예산”, “모델 fallback”를 따로 만드는데, 이 안은 이 셋이 실제로 연결된다는 걸 이해하고 있습니다.  
특히 아래 연결은 맞습니다.

- 토큰 사용량 → USD 환산
- USD 환산 → 예산 소진율 계산
- 소진율/정책 → 모델 자동 전환 및 차단
- 전환/재시도 → 추가 비용 추적

이 구조가 맞아야 운영이 됩니다.

### 2) USD를 기준축으로 둔 점

매우 타당합니다. 토큰 기준 예산은 운영자 입장에서 직관성이 낮고, 멀티벤더 환경에서는 더더욱 불리합니다.  
다만 “USD 기반”이라고 해서 실제 청구액과 내부 계산액이 같아지는 것은 아니므로, **내부 원가(estimated cost)** 와 **실청구/크레딧(ledger/billing)** 를 분리해야 합니다. 이건 장점이자 동시에 현재 설계의 가장 큰 보완 포인트입니다.

### 3) 단계적 제한(80/100/120%) 아이디어

운영 관점에서 괜찮습니다. 무조건 차단보다 훨씬 현실적입니다.  
특히 100%에서 저비용 모델 전환, 120%에서 차단은 실무적으로 납득 가능합니다.

### 4) LLM Judge를 전수평가가 아니라 샘플링으로 둔 점

좋습니다. 품질평가 체계를 억지로 전수화하지 않고 운영가능성을 고려했습니다.  
다만 샘플링 기준과 gold set 보정 체계가 없으면 “숫자는 나오는데 신뢰는 낮은” 시스템이 되기 쉽습니다.

## 2. 핵심적으로 아쉬운 부분

## A. “비용”과 “청구/크레딧”이 한 모델 안에서 섞여 있음

현재 문서에는 아래가 한 묶음처럼 보입니다.

- token_usage_logs 기반 내부 추정 비용
- 벤더 Admin API 기반 실제 잔여 크레딧
- 수동 입력 기반 잔액
- Workers AI 같은 역산치

이 네 개는 성격이 다릅니다.

### 왜 위험한가

같은 대시보드에 “총 크레딧 / 사용량 / 잔여”를 보여줄 때, 사용자는 이것을 **동일 정밀도의 숫자**로 받아들입니다.  
그런데 실제로는

- 어떤 값은 실청구 기반
- 어떤 값은 계약 크레딧 기반
- 어떤 값은 추정치
- 어떤 값은 관리자 입력치

일 수 있습니다.

즉, 지금 설계대로면 **운영 편의성은 높지만 재무 신뢰성은 떨어질 수 있습니다.**

### 보완 의견

데이터 모델을 최소 3계층으로 나누는 게 맞습니다.

- **Usage Ledger:** 호출 로그, 토큰, 사용자, 용도, 모델, latency
- **Cost Estimation Ledger:** 단가표 기준 추정 USD
- **Billing/Credit Ledger:** 벤더 API 또는 수동 입력 기반 실제 크레딧/청구 잔액

그리고 UI에서 명확히 구분해야 합니다.

- “예상 사용비용”
- “실제 벤더 잔여 크레딧”
- “관리자 입력 기준 잔액”

이걸 안 나누면 나중에 “왜 대시보드 잔여랑 실제 청구서가 다르냐”는 문제가 반드시 납니다.

## B. SSOT가 “모델별 단가표” 하나로 끝나지 않음

문서에 “모델별 단가 테이블이 SSOT”라고 했는데, 절반만 맞습니다.

실제 SSOT는 최소 4개입니다.

- **모델 카탈로그**: provider, model_id, capability, availability
- **가격 카탈로그**: input/output/cache/tool/image 등 단가 버전
- **정책 카탈로그**: fallback 우선순위, 금지 모델, budget tier별 허용모델
- **계약/크레딧 카탈로그**: tenant별 구매 크레딧, 계약기간, 공급자별 한도

단가표만 SSOT로 두면 환산은 되지만 정책 집행은 못 합니다.

### 특히 빠진 것

- 가격 유효기간 시작/종료일
- 모델 alias 변경 이력
- provider별 과금 단위 차이
- cached input / reasoning / tool call / image / audio 같은 비토큰성 과금 항목
- 부가비용(egress, vector, storage, rerank 등) 포함 여부

즉, **“토큰 x 단가”만으로는 2026년형 AI API 운영원가를 충분히 설명하기 어렵습니다.**

## C. 예산 계층 구조가 불명확함

B2에서 “전역 + 사용자별 개인 할당액”을 두었는데, 실제 운영에서는 보통 아래 계층이 필요합니다.

- 전사/시스템
- tenant/조직
- 프로젝트/워크스페이스
- 사용자
- 용도(채팅/분석/배치)
- 모델군 또는 공급자

현재 안은 전역과 사용자만 있어 보이는데, 이러면 실제 비용 통제 포인트가 너무 거칩니다.

### 문제 예시

한 사용자가 개인 예산은 남아 있어도,
- 특정 프로젝트 예산은 초과했을 수 있고
- 특정 공급자 예산은 막아야 할 수 있고
- 배치 작업은 허용, 실시간 채팅은 제한해야 할 수 있습니다.

따라서 **예산 객체를 “누구(user)” 기준만이 아니라 “무엇을 위해(purpose/project)” 기준으로도 가져가야** 합니다.

## D. 100% 초과 시 “저비용 모델 자동 전환”은 생각보다 위험함

아이디어는 좋지만, 그냥 붙이면 품질/계약/보안 문제를 만듭니다.

### 왜 그런가

비용이 낮다고 해서 대체 가능한 모델이 아닐 수 있습니다.

- JSON strict output 필요한 업무
- 긴 context 필요한 업무
- tool calling 안정성이 중요한 업무
- 안전성/정책 준수가 중요한 업무
- 특정 벤더만 허용되는 데이터 경계

이런 경우 저비용 모델로 자동 전환하면 실패율이 오르고, 오히려 재시도로 비용이 더 늘 수 있습니다.

### 보완 의견

“100% → 저비용 모델 전환”은 단일 정책이 아니라 아래 형태가 적절합니다.

- **capability class별 대체 정책**
  - high-reliability structured output
  - general chat
  - batch summarization
  - retrieval answer
- **업무 중요도별 정책**
  - soft-degrade 허용
  - degrade 금지
- **tenant policy override**
- **admin exception**

즉, fallback은 provider 체인이 아니라 **정책 기반 라우팅 테이블**로 가야 합니다.

## E. C1/C2의 “시스템 자동 모델 선택”은 좋아 보이지만 설명가능성이 부족함

사용자 선택권을 줄이고 시스템 자동 선택으로 가는 것은 운영상 장점이 있지만, 그만큼 내부적으로는 결정 로직이 투명해야 합니다.

현재 요구사항만 보면 시스템이 다음 중 무엇을 우선하는지 불명확합니다.

- 비용
- 품질
- latency
- quota availability
- JSON/tool-call compatibility
- 지역/보안 정책
- budget state

이 로직이 없으면 나중에 운영자가 “왜 이 요청이 이 모델로 갔는지” 설명할 수 없습니다.

### 반드시 필요한 것

모든 호출에 대해 **routing decision log** 를 남겨야 합니다.

예:
- selected_provider
- selected_model
- candidate_chain
- reason_code
- budget_state
- fallback_count
- policy_version

이게 없으면 디버깅도 안 되고 감사도 안 됩니다.

## F. LLM Judge 샘플링은 방향은 맞지만 아직 운영설계가 약함

현재 C3는 “N% 랜덤 샘플링” 정도인데, 이것만으로는 평가체계가 약합니다.

### 부족한 점

- 어떤 응답을 샘플링할지 기준 없음
- judge prompt/version 관리 없음
- human review calibration 없음
- 결과를 routing policy와 어떻게 연결할지 없음
- 도메인별 품질 기준 차이 없음

### 보완 의견

최소한 아래로 가야 합니다.

- 랜덤 샘플 + 위험기반 샘플 혼합
  - 긴 응답
  - fallback 발생 건
  - 재시도 후 성공 건
  - structured output failure recover 건
  - high-cost request
- judge versioning
- 월간 human spot-check
- 품질지표를 모델군별로 분리
  - faithfulness
  - format compliance
  - task success proxy
- 일정 임계치 이하일 때만 routing weight 조정

즉, Judge는 “모니터링”으로 시작해야지 바로 “자동 라우팅 제어”까지 연결하면 과제 범위가 너무 커집니다.

## G. 남용 감지(C5)는 너무 추상적임

“단시간 대량 호출, 반복 동일 요청” 수준은 출발점이지 설계가 아닙니다.

실제로는 다음이 필요합니다.

- 탐지 단위: user / tenant / API key / IP / session / project
- 룰 타입: rate / burst / repetition / anomaly / exfiltration-like pattern
- 대응 타입: alert / soft throttle / hard block / captcha-like verification / admin review
- 화이트리스트/예외
- false positive 처리

특히 내부 기업 환경이면 “남용”과 “정상 배치작업”을 구분해야 합니다.  
지금 상태로는 기능명만 있고 운영정책이 없습니다.

## 3. 개발계획으로 볼 때 가장 큰 리스크

### 1) 범위가 넓은데 MVP 컷이 안 보임

지금 안은 사실상 다음을 한 번에 하겠다는 것입니다.

- 멀티벤더 비용 집계
- 예산 정책 엔진
- 사용자 UI
- 관리자 대시보드
- 동적 fallback
- 품질평가 샘플링
- 자동 재시도
- 남용 감지

이건 제품 1개 분량이 아니라 **운영 플랫폼 2~3개 분량**입니다.

### 2) 정책 엔진 없이 UI부터 붙을 가능성

문서상으로는 사용자 프로필 카드, 채팅 잔여 배너, 관리자 대시보드가 빨리 보이는데,  
실제 성공 여부는 UI가 아니라 아래에 달려 있습니다.

- 정산 기준
- 정책 우선순위
- 예외 처리
- ledger 구조
- event logging

여기가 먼저입니다.

### 3) 벤더 연동 가능성에 대한 현실성 차이

A2에서 Anthropic은 자동, OpenAI/Google은 “가능하면 연동, 아니면 수동”인데,  
이건 요구사항 차원에서는 솔직하지만, 제품 설계 차원에서는 **자동/반자동/수동이 섞인 상태를 정식 모델로 받아들여야** 합니다.

즉 “미연동 시 수동 입력”은 임시방편이 아니라 제품 기능이어야 합니다.

- 입력 주체
- 증빙
- 변경 이력
- 유효기간
- 승인 프로세스

이게 없으면 관리자 수동 입력은 곧 신뢰성 하락 지점이 됩니다.

## 4. 객관적 검토 의견: 무엇을 먼저 고쳐야 하는가

### 1순위: 데이터/정산 모델을 먼저 고정

개발 전에 아래 엔터티를 먼저 확정하는 게 맞습니다.

- usage_event
- cost_estimate
- billing_balance
- budget_policy
- budget_period
- routing_policy
- routing_decision
- retry_event
- quality_sample
- abuse_signal

이 스키마가 먼저 안 나오면 기능 구현이 각자 따로 생깁니다.

### 2순위: 정책 우선순위를 명시

정책 충돌 순서를 정해야 합니다.

예시:
1. 보안/규제 제한
2. 강제 금지 모델 정책
3. 예산 상태
4. 기능 적합성(JSON/tool/context)
5. 공급자 가용성
6. 비용 최적화
7. 품질 통계 기반 미세조정

이 순서가 없으면 fallback과 budget degrade가 충돌합니다.

### 3순위: MVP 범위를 절반으로 줄이기

현실적으로는 아래 정도가 1차입니다.

**Phase 1**
- usage ledger
- USD estimate
- admin budget setup
- 80/100/120 alert + hard policy
- routing decision log
- admin dashboard

**Phase 2**
- user visibility UI
- provider credit dashboard
- manual credit inputs with audit
- soft model degrade policy

**Phase 3**
- LLM Judge sampling
- quality-based retry
- abuse detection

이 순서가 더 맞습니다.

## 5. 각 영역별 세부 검토의견

## A. 비용 관리

### 평가
방향은 좋지만, **추정 비용과 실제 잔액의 혼합표시 리스크**가 큽니다.

### 검토의견
- A1은 꼭 필요합니다.
- A2/A3는 구현 가능성보다 **데이터 신뢰도 라벨링**이 더 중요합니다.
- A4는 필수이지만 토큰 외 과금요소 반영 범위를 명확히 해야 합니다.
- A5는 사용자 경험상 좋지만, 숫자 신뢰성이 확보된 뒤 붙이는 게 맞습니다.

### 추가 권고
사용자에게는 “잔여 예산”보다  
**현재 기간 사용률, 정책 상태, 제한 예상 동작**을 보여주는 쪽이 낫습니다.  
예: “예산 96% 사용, 초과 시 저비용 모드 전환 예정”

## B. 예산 관리

### 평가
이 영역이 사실상 핵심입니다. 그러나 현재는 정책 정의가 덜 끝났습니다.

### 검토의견
- B1은 타당
- B2는 사용자별만으로 부족
- B3는 좋음. 다만 overlapping period 허용 여부 결정 필요
- B4는 유용하지만 capability-aware 정책으로 바꿔야 함
- B5는 매우 중요. 다만 “용도” taxonomy를 먼저 표준화해야 함

### 추가 권고
용도 분류를 사전에 고정하세요.  
예:
- chat
- analysis

- batch
- eval
- agent-tool-run

이 분류 없으면 나중에 정산리포트가 쓸모없어집니다.

## C. API 관리

### 평가
가장 매력적이지만, 동시에 가장 복잡합니다.

### 검토의견
- C1/C2는 routing policy engine 수준으로 재정의 필요
- C3는 관측 기능으로 먼저 도입하는 게 적절
- C4는 “기본 검증 실패 시 재시도”가 아니라 **retry taxonomy**가 먼저 필요
- C5는 운영보안 정책과 연결해야 합니다

### C4 관련 특히 중요
재시도는 아래로 분리해야 합니다.

- transport failure retry
- provider quota failover
- structured output recovery retry
- empty response retry
- semantic low-quality retry

이건 각각 비용/성공률/위험이 다릅니다. 한 정책으로 묶으면 안 됩니다.
